# 배포와 기술 방향

## 제품 방향

초기에는 네 명이 쓰는 작은 웹앱이므로 운영 복잡도를 낮춘다. 가장 먼저 관리자가 짜둔 기본 일정을 화면에 띄우고, 그 다음 후보 장소 제안, 관리자 승인, 회계 기능을 붙인다.

## 추천 스택

### 프론트엔드

- Next.js 또는 React
- TypeScript
- 모바일 우선 반응형 UI
- 간단한 상태 관리는 React 상태 또는 Zustand

### 스타일

- Tailwind CSS 또는 CSS Modules
- 하단 탭, 타임라인, 리스트 중심의 조용한 UI

### 데이터

- Supabase를 기본 데이터 저장소로 사용한다.
- 후보 장소, 추천, 코멘트, 방문 확인, 회계 데이터를 Supabase에 저장한다.
- 기본 일정은 seed 데이터로 넣고, 필요하면 Supabase에서 관리한다.
- 실시간 동기화가 꼭 필요하지 않다면 일반 CRUD만으로 충분하다.
- 환율 자동 계산은 서버 API 또는 Render 백엔드에서 처리한다.

## 인증 방식

- 멤버 선택 후 사용한다.
- 지환, 지완, 지원은 이름 선택만으로 입장한다.
- 주환은 관리자 접근 코드를 입력해야 관리자 권한을 사용한다.
- 관리자 접근 코드는 `ADMIN_ACCESS_CODE` 환경변수로 관리한다.
- 주환만 기본 일정 수정, 후보 승인, 방문 완료 처리가 가능하다.
- 멤버는 후보 장소 추가, 추천, 코멘트, 회계 입력이 가능하다.

여행 앱 특성상 소셜 로그인은 나중으로 미룬다. 링크가 외부로 노출될 수 있으므로 민감 정보는 최소한만 넣는다.

## 배포

- Vercel에 웹앱 배포
- Supabase에 데이터 저장
- Render에 필요한 서버 API 또는 환율 API 프록시 배포
- 환경 변수로 Supabase URL과 anon key 관리
- 환경 변수로 `ADMIN_ACCESS_CODE`, 환율 API 키 관리

## 보안 메모

- 여권 번호, 카드 번호, 주민등록번호 등 민감 정보는 저장하지 않는다.
- 예약 번호와 숙소 주소는 MVP 이후 필요할 때만 추가한다.
- 공개 저장소로 운영한다면 실제 여행 데이터는 코드에 넣지 않는다.
- 관리자 접근 코드는 저장소나 문서에 평문으로 남기지 않는다.
- 환율 API 키도 환경변수로만 관리한다.
